Chapter 1

ウェブアクセシビリティとは

定義

ウェブアクセシビリティとは、年齢・障害の有無・使用環境に関わらず、誰もがWebを利用できる状態のことです。

似た言葉の「ユーザビリティ」は使いやすさ全般を指しますが、アクセシビリティは特に使えるかどうか（利用可能性）に焦点を当てています。

誰のための話か

障害・状況 Webで困ること 対応例
視覚障害 画像の内容がわからない、スクリーンリーダーで読めない alt属性、適切な見出し構造
色覚特性 色だけで伝えられた情報がわからない コントラスト確保、色以外の情報も併用
肢体不自由 マウス操作ができない キーボードだけで操作できる設計
聴覚障害 動画の音声内容がわからない 字幕・テキストトランスクリプトの提供
認知・学習障害 複雑な操作、難しい言葉が理解しにくい わかりやすい文章、明確なラベル
高齢者 小さい文字が見えない、操作しにくい 十分なフォントサイズ、大きなタップ領域
💡 ポイント
障害のある人だけの話ではありません。スマートフォンのユーザー、屋外の明るい場所での利用、一時的な怪我、非ネイティブスピーカーなど、誰もが恩恵を受ける設計です。

なぜ今、必要なのか

  • 障害者差別解消法（2024年4月改正）：民間事業者にも「合理的配慮」の提供が義務化。WebサイトのアクセシビリティはこのDEの対象になりうる
  • 公共機関は対応が必須：国・自治体のサイトはJIS X 8341-3 レベルAAへの準拠が求められている
  • 就職・仕事での需要増：クライアントから求められるスキルになっている
  • SEO・UXとも連動：適切な見出し構造・alt属性・テキスト情報はSEOにも直結する

日本のJIS規格はWCAGをベースにしています。つまりWCAGを理解すればJISも理解できます。

WCAG（ウェブ・コンテンツ・アクセシビリティ・ガイドライン）

W3C（ウェブ技術の国際標準化団体）の中のWAI（Web Accessibility Initiative）が策定する国際ガイドライン。

バージョン年特徴
WCAG 2.02008年現行JISのベース。レベルA+AAで38項目
WCAG 2.12018年モバイル・認知障害への対応を追加。レベルA+AAで50項目
WCAG 2.22023年最新版。タップ領域・フォーカス表示などを強化。レベルA+AAで50項目（合計86項目）

JIS X 8341-3（日本の規格）

  • 現行：JIS X 8341-3:2016（WCAG 2.0ベース）
  • ⚠️ 2026年度に改正予定（WCAG 2.2対応へ）— 「廃止」ではなく「アップデート」
  • 公共機関は適合レベルAAへの準拠が求められている

WAIC（ウェブアクセシビリティ基盤委員会）

日本のウェブアクセシビリティにおいて最も重要な委員会です。JIS X 8341-3の理解・普及促進、WCAG日本語訳の公開などを行っています。

📌 WAIC が提供している主なコンテンツ
  • WCAG 2.2 日本語訳：原文を日本語で読める
  • JIS X 8341-3:2016 解説：JIS規格の読み解き方
  • 達成基準 早見表（レベルA & AA）：チェックリストとして使える
  • ウェブアクセシビリティ方針策定ガイドライン：実務で方針ページを作るときに

👉 https://waic.jp/

その他、日本の関連ガイドライン

  • デジタル庁「ウェブアクセシビリティ導入ガイドブック」：初心者向け・図解豊富
  • 総務省「みんなの公共サイト運用ガイドライン（2024年版）」：公共機関向け手順書
Chapter 3

WCAGの4原則と適合レベル

4原則（POUR）

原則英語意味例
知覚可能 Perceivable 情報をユーザーが認識できること 画像にalt、動画に字幕
操作可能 Operable UIを操作できること キーボード操作、十分な時間
理解可能 Understandable 情報・操作方法が理解できること lang属性、明確なエラー表示
堅牢 Robust 支援技術に対応できること 正しいHTMLマークアップ、ARIA

適合レベル

レベル内容基準数（WCAG 2.2）
A 最低限の対応。これができていないと一部ユーザーが利用不可能になる 31項目
AA ★ 推奨レベル。公共機関サイトはこれが求められる A+AAで50項目
AAA 最高レベル。すべてのコンテンツへの適用は難しい場合もある 合計86項目
💡 「準拠」と「一部準拠」
公共サイトのフッターなどに「JIS X 8341-3:2016 レベルAAに準拠」と記載されているのを見たことがあるかもしれません。すべての基準を満たせば「準拠」、1つでも満たせなければ「一部準拠」と表記します。
Chapter 4

実装ポイント

alt属性（代替テキスト）

画像の意味をテキストで伝える属性です。スクリーンリーダーはこの内容を読み上げます。

❌ NG — alt属性なし
<!-- ファイル名が読み上げられてしまう -->
<img src="banner.jpg">
✅ OK — 内容を説明
<img src="banner.jpg"
     alt="春の新商品キャンペーン告知">
❌ NG — ファイル名や「画像」はダメ
<img src="logo.svg" alt="logo.svg">
<img src="photo.jpg" alt="画像">
✅ OK — 装飾画像はalt=""（空）
<!-- 情報を持たない装飾画像は空にする -->
<img src="deco.png" alt="">
💡 alt=""（空）と alt属性なし の違い
alt="" はスクリーンリーダーが画像の存在自体をスキップします。一方、alt属性がない場合はファイル名などを読み上げてしまうことがあります。装飾画像には必ず alt="" を書きましょう。

見出し構造（h1〜h6）

見出しはスクリーンリーダーユーザーがページを「ナビゲート」するために使います。見た目のサイズを調整したいだけのために見出しタグを選ぶのはNGです。

❌ NG — 見た目だけで選んでいる
<!-- 大きく見せたいだけでh1を使う -->
<h1>会社概要</h1>
<h1>事業内容</h1>
<h1>お問い合わせ</h1>
✅ OK — 階層構造に沿って使う
<h1>株式会社〇〇</h1>
  <h2>会社概要</h2>
    <h3>沿革</h3>
  <h2>事業内容</h2>
  <h2>お問い合わせ</h2>
⚠️ よくある間違い
h1 はページ内に1つだけが基本です。見出しレベルは飛ばさないようにしましょう（h2 の次に h4 など）。

lang属性

スクリーンリーダーが正しい言語で読み上げるために必要な属性です。

HTML
<!-- 日本語ページには lang="ja" を必ず指定 -->
<html lang="ja">

<!-- ページ内に英語が含まれる場合は部分的に指定も可能 -->
<p>詳細は <span lang="en">Privacy Policy</span> をご確認ください。</p>

フォームと label 要素

label 要素を使ってフォーム部品と説明を関連づけます。スクリーンリーダーはラベルを読み上げて、何を入力するかをユーザーに伝えます。

❌ NG — placeholderだけに頼っている
<!-- 入力後はplaceholderが消えて何の項目かわからなくなる -->
<input type="text" placeholder="お名前">
✅ OK — for属性とidで紐づける
<label for="name">お名前</label>
<input type="text" id="name"
       placeholder="山田 太郎">

カラーコントラスト比

文字と背景の明暗差（コントラスト比）が不足すると、見えにくくなります。WCAGレベルAAでは通常テキストで4.5:1以上が必要です。

❌ NG — コントラスト不足
/* コントラスト比 約2.3:1（不十分） */
.text {
  color: #aaaaaa;
  background: #ffffff;
}
✅ OK — 4.5:1以上を確保
/* コントラスト比 約7:1（十分） */
.text {
  color: #595959;
  background: #ffffff;
}
💡 大きな文字・ロゴは基準が異なる
18pt以上の大きなテキストや太字テキスト（14pt以上）は3:1以上でOKです。ロゴなどに含まれる文字は基準の対象外です。

フォーカススタイル

キーボードでTabキーを押して移動するとき、どこにフォーカスがあるかが視覚的に見えることが必要です。outline: none で消してしまうのはアクセシビリティ違反です。

❌ NG — フォーカスを消している
a:focus,
button:focus {
  outline: none; /* ← 絶対にダメ */
}
✅ OK — 視認できるスタイルを設定
a:focus-visible,
button:focus-visible {
  outline: 3px solid #005fcc;
  outline-offset: 2px;
}
💡 :focus-visible を使う
:focus ではなく :focus-visible を使うと、マウスクリック時にはフォーカスリングを表示せず、キーボード操作時にのみ表示させることができます。

.sr-only（スクリーンリーダー専用テキスト）

視覚的には非表示にしつつ、スクリーンリーダーには読み上げてもらいたいテキストに使うCSSパターンです。

CSS
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}
HTML — 使用例
<!-- アイコンだけのボタン — 視覚的には✕アイコンだけ表示 -->
<button type="button">
  ✕
  <span class="sr-only">モーダルを閉じる</span>
</button>

<!-- 「もっと見る」ボタン — スクリーンリーダーには何のリンクか補足 -->
<a href="news.html">
  もっと見る
  <span class="sr-only">（ニュース一覧）</span>
</a>
⚠️ display:none や visibility:hidden との違い
display: none はスクリーンリーダーにも読まれません。.sr-only は「視覚的に隠しつつ、スクリーンリーダーには伝える」ための専用テクニックです。

WAI-ARIAについて（参考）

📎 HALベーカリー＆カフェの教材でも出てきましたね

ハンバーガーメニューで使った aria-label / aria-controls / aria-expanded / aria-hidden、アコーディオンで使った aria-expanded の動的切り替えがこれにあたります。

WAI-ARIAはHTMLだけでは支援技術に伝えられない情報を補うための仕様です。詳しくはWAICの「WAI-ARIA オーサリング・プラクティス（日本語訳）」を参照してください。

チェック・検証方法

自動チェックツール（ブラウザ拡張）

ツール特徴
axe DevTools Chrome拡張。開発者ツールに統合。問題箇所をコードと紐づけて表示
WAVE Chrome拡張。ページ上にアイコンを重ねて視覚的に問題を確認できる
Lighthouse Chrome DevToolsに内蔵。アクセシビリティスコアをまとめて確認
⚠️ 自動チェックで検出できるのは全体の30〜40%程度
ツールで問題なしでも、キーボード操作の確認・実際にスクリーンリーダーで試すことが大切です。

カラーコントラストチェック

  • WebAIM Contrast Checker（webaim.org）— 色コードを入力してコントラスト比を確認
  • Figma プラグイン「Contrast」 — デザイン段階でのチェックに

スクリーンリーダーで体験してみる

  • macOS：VoiceOver（標準搭載）— ⌘ + F5 で起動
  • Windows：NVDA（無料・ダウンロード）
  • iPhone：VoiceOver（設定 → アクセシビリティ）
  • Android：TalkBack（設定 → ユーザー補助）
