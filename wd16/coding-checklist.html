コーディングチェックリスト

このチェックリストは、HTMLとCSSコーディングで陥りやすいミスや、見落としがちなポイントをまとめたものです。
コーディング後、必ず確認してください。

HTMLコーディング時に気をつけること

1. HTML基本構造

<html>タグのlang属性は、ページの主要言語を示します。日本語サイトの場合は必ずlang="ja"を指定してください。
❌ 悪い例
<html lang="en">  <!-- 日本語コンテンツなのに英語指定 -->
✅ 良い例
<html lang="ja">
なぜ重要？ スクリーンリーダーが正しい発音で読み上げるため、検索エンジンが言語を正しく認識するために必要です。
リセットCSS（destyle.css、normalize.cssなど）は、ブラウザごとのデフォルトスタイルの違いをなくすためのものです。必ず独自のCSSより先に読み込んでください。
❌ 悪い例
<link rel="stylesheet" href="css/style.css">
<link rel="stylesheet" href="css/destyle.css">
✅ 良い例
<link rel="stylesheet" href="css/destyle.css">
<link rel="stylesheet" href="css/style.css">
理由： 後に読み込まれたCSSが優先されます。リセットCSSを後に読み込むと、独自スタイルが上書きされてしまいます。
開始タグと終了タグのペアが正しく対応しているか確認してください。特に<section>、<div>、<ul>などのネストが深い場合は注意が必要です。
❌ 悪い例
<section class="menu">
  <h2>クロワッサン</h2>
  <p>¥380</p>
<!-- </section>の閉じタグがない -->
確認方法： ブラウザの開発者ツールでHTML構造を確認する、HTMLバリデーターでチェックする、エディタのシンタックスハイライトを確認する。

2. 見出しタグ（最重要）

<h1>はページの最上位の見出しで、1ページに必ず1つだけ配置します。通常はサイトロゴやページタイトルに使用します。
✅ 良い例
<header>
  <h1><img src="logo.svg" alt="Boulangerie HAL"></h1>
  <nav>...</nav>
</header>
重要： <h1>が0個、または2個以上あるのはNGです。SEOとアクセシビリティの観点から必須です。
見出しは階層構造を表します。h2の次にh4を使うなど、レベルを飛ばしてはいけません。
❌ 悪い例
<h2>メニュー</h2>
<h4>パンの種類</h4>  <!-- h3を飛ばしている -->
✅ 良い例
<h2>メニュー</h2>
<h3>パンの種類</h3>
補足： 見出しのサイズ（見た目）はCSSで調整します。HTML構造は論理的な階層を表現します。
<section>は独立したコンテンツのまとまりを表します。必ず見出し（h2〜h6）が必要です。
❌ 悪い例
<section class="about">
  <p>私たちについて</p>  <!-- 見出しがない -->
</section>
✅ 良い例
<section class="about">
  <h2>私たちについて</h2>
  <p>説明文...</p>
</section>
見出しが不要な場合： 見出しが必要ない場合は<section>ではなく<div>を使用してください。
セクションのタイトルや項目名など、見出しとして機能するテキストは必ず見出しタグを使用してください。<p>タグで書いてはいけません。
❌ 悪い例
<p class="section-title">メニュー</p>
<p class="item-name">クロワッサン</p>
✅ 良い例
<h2 class="section-title">メニュー</h2>
<h3 class="item-name">クロワッサン</h3>
判断基準： 「これは何のセクションか」「これは何の項目か」を示すテキストは見出しです。CSSでサイズを調整できるので、見た目ではなく役割で判断してください。

3. セマンティックHTML

ページの主要なコンテンツは<main>タグで囲みます。<header>、<footer>、<nav>などは含めません。
✅ 良い例
<body>
  <header>...</header>
  <main>
    <section class="hero">...</section>
    <section class="about">...</section>
    <section class="menu">...</section>
  </main>
  <footer>...</footer>
</body>
ルール： <main>は1ページに1つだけです。
セマンティックタグを使うことで、HTMLの構造が明確になり、アクセシビリティとSEOが向上します。
  • <header>: サイトヘッダーやセクションのヘッダー
  • <footer>: サイトフッターやセクションのフッター
  • <nav>: ナビゲーションメニュー
  • <section>: 独立したコンテンツのまとまり（見出し必須）
  • <article>: 単独で完結する記事やコンテンツ
ポイント： header、footer、nav、sectionあたりは使い方を覚えましょう。それ以外は、自信がない場合は無理にセマンティックタグを使わず、<div>を使う方が安全とも言えます。間違った使い方の方が問題になります。
<section>には必ず見出しが必要です。見出しがない場合は<div>を使用してください。
❌ 悪い例
<section class="container">  <!-- 見出しがない -->
  <div class="item">...</div>
</section>
✅ 良い例
<div class="container">
  <div class="item">...</div>
</div>

4. 画像とアクセシビリティ

alt属性は必須です。画像が表示されない場合や、スクリーンリーダーを使用するユーザーのために必要です。
❌ 悪い例
<img src="bread.jpg">  <!-- alt属性がない -->
✅ 良い例
<img src="bread.jpg" alt="焼きたてのクロワッサン">
alt属性には画像の内容を具体的に説明するテキストを入れます。ファイル名をそのまま使うのはNGです。
❌ 悪い例
<img src="croissant.jpg" alt="croissant">
<img src="logo.svg" alt="logo">
✅ 良い例
<img src="croissant.jpg" alt="焼きたてのクロワッサン">
<img src="logo.svg" alt="Boulangerie HAL">
装飾目的だけの画像（背景パターン、デザイン要素など）は、alt=""（空）にすることで、スクリーンリーダーが読み飛ばします。
✅ 良い例
<img src="decoration.svg" alt="">  <!-- 装飾画像 -->
判断基準： 「この画像がなくても内容が理解できるか？」→ Yes なら装飾画像（alt=""）

5. リンクとボタン

ページ内リンク（アンカーリンク）を使う場合、リンク先の要素に対応するid属性が必要です。
❌ 悪い例
<a href="#menu">メニューへ</a>

<section class="menu">  <!-- id属性がない -->
  <h2>メニュー</h2>
</section>
✅ 良い例
<a href="#menu">メニューへ</a>

<section class="menu" id="menu">  <!-- id属性を追加 -->
  <h2>メニュー</h2>
</section>
確認方法： リンクをクリックして、実際に目的の場所にジャンプするか確認してください。
ページ遷移しない操作（モーダルを開く、メニューを開閉するなど）は<a>ではなく<button>を使います。
❌ 悪い例
<a href="#" onclick="openModal()">詳細を見る</a>
✅ 良い例
<button onclick="openModal()">詳細を見る</button>
使い分け： <a>はページ遷移、<button>は操作・アクション
HTML5では<a>の中にブロック要素を入れることができますが、単純なテキストリンクの場合は<p>は不要です。
❌ 不必要な例
<a href="#home"><p>ホーム</p></a>
✅ 良い例
<a href="#home">ホーム</a>
使って良い場合： カード型リンクなど、複数の要素をまとめてリンクにする場合は適切です。

6. ファイルパス

画像パスの書き方（./images/、images/、../images/）が混在していないか確認してください。
❌ 悪い例（混在）
<img src="images/hero.jpg" alt="店舗外観">
<img src="./images/bread.jpg" alt="パン">
<img src="../images/shop.jpg" alt="店内の様子">
✅ 良い例（統一）
<img src="./images/hero.jpg" alt="店舗外観">
<img src="./images/bread.jpg" alt="パン">
<img src="./images/shop.jpg" alt="店内の様子">
推奨： HTMLファイルから見た相対パスで統一するのが分かりやすいです。
ファイル名は半角英数字と記号（ハイフン、アンダースコア）のみを使用してください。日本語や空白はサーバーやブラウザによって問題が起きる可能性があります。
❌ 悪い例
images/写真素材_BoulangerieHAL/パン001.jpg
images/Rectangle 14.png
✅ 良い例
images/boulangerie-hal/pain001.jpg
images/rectangle-14.png
重要： 特にサーバーにアップロードする際、日本語ファイル名は文字化けやリンク切れの原因になります。

7. コピーライト表記

コピーライト表記は<small>タグを使って以下の形式で記載します。基本は「©マーク + 最初の発行年 + 著作権者名」です。
✅ 基本形式
<p><small>© 2025 Boulangerie HAL</small></p>
✅ 最初の発行年を明記する場合
<p><small>© 2020 Boulangerie HAL</small></p>
ポイント： 著作権は最初に発行（公開）された年から発生します。更新年は記載しなくても問題ありません。リニューアルの場合も、最初の発行年を記載するのが基本です。

CSSコーディング時に気をつけること

1. セレクタの書き方

section.newsのように要素型とクラスを併記すると、詳細度が無駄に高くなります。クラスだけで十分です。
❌ 悪い例
section.news { }
div.contents { }
p.date { }
✅ 良い例
.news { }
.contents { }
.date { }
理由： 詳細度が高いと、後から上書きするのが難しくなります。また、要素型を指定すると以下の問題があります：
  • HTML構造が変わったとき（sectionからdivに変更など）にCSSも修正が必要
  • 同じ見た目でも要素が違う場合（<a>のボタンと<button>のボタン）にスタイルを共有できない
  • 他のページで再利用しにくい
headerやfooterなどの要素セレクタに直接スタイルを書くと、再利用性が低くなります。クラスを付けて管理しましょう。
❌ 悪い例
header {
  background-color: #333;
  padding: 20px;
}

footer {
  background-color: #333;
  padding: 20px;
}
✅ 良い例
.site-header {
  background-color: #333;
  padding: 20px;
}

.site-footer {
  background-color: #333;
  padding: 20px;
}
例外： リセットCSS的な基本スタイル（body { font-family: sans-serif; }など）は要素セレクタでOKです。
セレクタチェーンが長いと詳細度が高くなり、メンテナンスが困難になります。3〜4階層を目安にしてください。
❌ 悪い例
.section .container .item .text .title {
  /* 5階層 */
}
✅ 良い例
.item-title {
  /* シンプル */
}
ヒント： クラス名で意味を表現できれば、深いネストは不要です。

2. コードの重複

同じスタイルを複数の場所で書いている場合は、共通クラスにまとめましょう。
❌ 悪い例
.news h2 {
  font-size: 32px;
  text-align: center;
  margin-bottom: 50px;
}

.menu h2 {
  font-size: 32px;
  text-align: center;
  margin-bottom: 50px;
}

.address h2 {
  font-size: 32px;
  text-align: center;
  margin-bottom: 50px;
}
✅ 良い例
.section-title {
  font-size: 32px;
  text-align: center;
  margin-bottom: 50px;
}
複数の要素で使用するスタイルは、共通クラスとして定義します。
✅ 良い例
/* 共通クラス */
.container {
  max-width: 1200px;
  margin: 0 auto;
  padding: 0 20px;
}

.section-padding {
  padding: 80px 0;
}
CSS変数（カスタムプロパティ）を定義したら、実際に使用しているか確認してください。
❌ 悪い例
:root {
  --primary-color: #333;
  --secondary-color: #666;
}

/* 変数を使わず直接色を指定 */
.header {
  background-color: #333;
}
✅ 良い例
:root {
  --primary-color: #333;
  --secondary-color: #666;
}

.header {
  background-color: var(--primary-color);
}

3. レイアウト

コンテナの幅指定にはmax-widthを使うことで、小さい画面でも適切に表示されます。
❌ 悪い例
.container {
  width: 1200px;  /* 画面幅が1200px未満だと横スクロールが出る */
}
✅ 良い例
.container {
  max-width: 1200px;  /* 画面幅に応じて縮小される */
  margin: 0 auto;
}
bodyや通常のコンテンツ要素に固定のheightを指定すると、コンテンツが増えたときに問題が起きます。
❌ 悪い例
body {
  height: 1080px;  /* コンテンツが増えると溢れる */
}

.card {
  height: 200px;  /* テキストが増えると溢れる */
}
✅ 良い例
body {
  min-height: 100vh;  /* 最小高さを指定 */
}

.card {
  /* heightは指定せず、paddingで調整 */
  padding: 20px;
}
例外： ヒーローセクションなど、意図的に画面全体の高さにする場合はheight: 100vh;（または100svh）などを使用してOKです。

4. 詳細度

!importantは最終手段です。適切なセレクタ設計で!importantなしで実現できます。
❌ 悪い例
.text {
  color: #333 !important;
  font-size: 16px !important;
}
✅ 良い例
.text {
  color: #333;
  font-size: 16px;
}
どうしても必要な場合： 外部ライブラリのスタイルを上書きする場合など、限定的に使用することもあります。
詳細度が高すぎると、後から上書きするのが困難になります。シンプルなセレクタを心がけましょう。
❌ 悪い例（詳細度が高い）
section.menu div.contents div.item {
  /* 詳細度: 0-0-3-3 */
}
✅ 良い例（詳細度が低い）
.menu-item {
  /* 詳細度: 0-0-1-0 */
}

5. ブラウザ互換性

比較的新しいCSS機能（animation-timeline: view()、:has()など）を使う場合は、対応ブラウザを確認してください。
確認方法： Can I useで対応ブラウザを確認できます。
注意： 学習課題では最新機能を試すのは良いですが、実案件では対応ブラウザの要件を確認してください。

6. 値の書き方

色の指定方法（hex、rgb、color name）が混在していないか確認してください。
❌ 悪い例（混在）
.header {
  background-color: #333;
  color: rgb(255, 255, 255);
  border-color: white;
}
✅ 良い例（統一）
.header {
  background-color: #333;
  color: #fff;
  border-color: #fff;
}
推奨： CSS変数を使うと、さらに管理しやすくなります。
補足： 透過（alpha値）を使う場合は、16進数（#333）とrgba（rgba(51, 51, 51, 0.8)）が混在することはあります。
存在しないプロパティや、無効な値を使っていないか確認してください。
❌ 悪い例
.box {
  padding: 20px auto;  /* paddingにautoは使えない */
}
✅ 良い例
.box {
  padding: 20px 0;
  margin: 0 auto;  /* 中央寄せならmarginでauto */
}
確認方法： ブラウザの開発者ツールで、無効なプロパティには警告が表示されます。

全般的なチェック項目

1. 命名規則

クラス名の命名規則が統一されているか確認してください。
  • ケバブケース: section-title
  • キャメルケース: sectionTitle
  • スネークケース: section_title
重要： どの記法を選んでも構いませんが、プロジェクト内で統一することが大切です。

2. プロジェクト構成

SCSSファイルをCSSにコンパイルする際、エラーが出ていないか確認してください。ブラウザで表示確認し、意図した通りにスタイルが適用されているかチェックしましょう。
SCSSでパーシャルファイル（_header.scss、_footer.scssなど）を使う場合、適切にインポートされているか確認してください。
✅ 良い例
// style.scss
@import 'reset';
@import 'variables';
@import 'header';
@import 'main';
@import 'footer';

3. デバッグ

ブラウザの開発者ツール（F12キー）でHTML構造を確認し、タグの閉じ忘れやネストの誤りがないかチェックしてください。
確認ポイント：
  • タグが正しく閉じられているか
  • ネストが正しいか
  • 意図しない要素が追加されていないか
ブラウザのコンソール（開発者ツール内）で、エラーや警告が表示されていないか確認してください。
よくあるエラー：
  • 画像ファイルが見つからない（404エラー）
  • JavaScriptの構文エラー
  • CSSファイルが見つからない
W3CのHTML Validatorで、HTMLの文法をチェックしてください。
チェック方法： W3C Markup Validation Serviceにアクセスし、HTMLファイルをアップロードまたはURLを入力してチェックします。

4. タイポチェック

コード全体を見直し、以下のようなミスがないか確認してください。
  • クラス名のスペルミス
  • 画像ファイル名のミス（logo.png を rogo.pngなど）
  • HTMLタグのスペルミス
  • CSSプロパティのスペルミス
  • altテキストやコンテンツの誤字
ヒント： エディタのスペルチェック機能を活用しましょう。

✅ すべての項目をチェックしましたか？

コーディング後は必ずこのチェックリストで確認してください。
最初は時間がかかりますが、慣れると自然に注意できるようになります。
良いコーディング習慣を身につけましょう！
