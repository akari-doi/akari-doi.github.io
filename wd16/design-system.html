デザインシステムと
デザイントークン

大きなプロダクトを複数人で一貫してつくるための仕組みと、その中核となる「値の管理」の考え方を学びます。

01 — Design System

デザインシステムとは？

デザインシステムとは、「デザインのルールブック＋再利用できる部品集」のことです。 Webサイトやアプリを複数ページ・複数人で作るときに起こる「バラつき」や「非効率」を防ぎます。

❌ デザインシステムなし

こんな問題が起きる

・AさんとBさんで作ったボタンの色が微妙に違う

・ページによってフォントサイズがバラバラ

・「青を少し変えたい」→ 全ページを手直し

・新メンバーが「どの色を使えばいい？」と毎回聞く

✅ デザインシステムあり

こうなる

・誰が作っても同じルールで統一される

・1か所変えると全体に反映される

・新メンバーもルールを見れば自分で判断できる

・デザインと開発が同じ言葉で話せる

デザインシステムに含まれるもの

🎨

スタイルガイド

色・フォント・余白・角丸などのルール一覧

🧩

コンポーネント

ボタン・カード・ナビゲーションなど再利用できる部品

🏷️

デザイントークン

値を名前で管理する仕組み（詳しくは次のセクションで）

📖

ガイドライン

いつ何を使うかの判断基準・使用例

♿

アクセシビリティ

誰でも使えるデザインにするためのルール

🔤

タイポグラフィ

見出し・本文・キャプションなど文字のルール

📌 まとめ：デザインシステムは「チームで一貫したデザインをつくるための共通言語と部品」。
有名な例：Google の Material Design、Apple の Human Interface Guidelines
02 — Design Token

デザイントークンとは？

デザイントークンとは、「デザインの値（色・サイズ・余白など）に名前をつけて管理する仕組み」です。 デザインシステムの中で「ルールの値を保管する箱」の役割を担います。

なぜ名前をつけるの？

例：ブランドカラーのピンク #e83d7a を300ページに直接使っていたとします。

「やっぱり色を変えたい」→ 300か所すべてを手動で書き換える必要があります。

デザイントークンを使えば、名前を1か所変えるだけで全体に反映されます。

/* ❌ 値を直接使っていると… */ button { background: #e83d7a; } input:focus { border-color: #e83d7a; } .icon { color: #e83d7a; } → 色を変えるときに全部探して書き換え… /* ✅ デザイントークンを使うと */ :root { --brand-pink: #e83d7a; ← 1か所だけ変えればOK } button { background: var(--brand-pink); } input:focus { border-color: var(--brand-pink); } .icon { color: var(--brand-pink); }

※ CSSのカスタムプロパティ（変数）と全く同じ考え方です！

03 — Token Structure

トークンの3層構造

実際のプロジェクトではトークンを3つの階層に分けて管理します。 以下の図はFigma公式ドキュメントの図解をもとにした構造です。

階層名 名前の例 役割
プリミティブ（原色） pink-600 = #e83d7a 純粋な値を登録する
セマンティック（意味） accent-hover = pink-600 「何に使う色か」を表す
コンポーネント search-border-active = accent-hover 「どの部品のどこか」を表す

▼ シナリオA：pink-600 がグローバルトークン

Global Token
pink-600 = #e83d7a
Semantic Token
accent-hover → pink-600
surface-highlight → pink-600
Component Token
search-border-active → accent-hover
habit-list-progress → accent-hover
button-primary → surface-highlight

UI（pink-600 適用時）

90%

▼ シナリオB：accent-hover のみ purple-600 に変更

Global Token
purple-600 = #7c3aed
（新たに追加）
pink-600 = #e83d7a
Semantic Token
accent-hover → purple-600
surface-highlight → pink-600
Component Token
search-border-active
habit-list-progress
button-primary（ピンクのまま）

UI（accent-hover のみ変更後）

90%

→ インプットの枠線・プログレスは紫に。ボタンはピンクのまま維持。

📌 3層構造のポイント：
Global（原色の値）→ Semantic（意味）→ Component（部品のどこか）
上流を変えると下流すべてが変わる。階層を分けることで「全体を変える/一部だけ変える」を自在にコントロールできます。

CSS変数との対応関係

HTMLとCSSを学んでいれば、CSSカスタムプロパティの考え方とまったく同じです。

Figma バリアブル
// Global Token pink-600 = #e83d7a // Semantic Token（エイリアス） accent-hover = pink-600 button-primary = accent-hover
CSS カスタムプロパティ
/* Global Token */ --pink-600: #e83d7a; /* Semantic Token */ --accent-hover: var(--pink-600); --button-primary: var(--accent-hover);
04 — Real World Example

実例：デジタル庁デザインシステム

デジタル
庁

デジタル庁デザインシステム β版（DADS）

日本政府のWebサービスを一貫して設計するための公式デザインシステム。無償で公開されており、誰でも参照・利用できます。

デジタル庁は、行政サービスのWebサイトを誰でも使いやすく、統一感のあるデザインで提供するために、 デザインシステムをオープンに公開しています。

含まれている内容

🎨

基本デザイン

カラー・タイポグラフィ・余白・アイコン・角丸など

🧩

コンポーネント

40種類以上のUIパーツをコードつきで公開

♿

アクセシビリティ

JIS X 8341-3（WCAG）への対応方針を明示

収録されているコンポーネントの例

ボタン インプットテキスト チェックボックス ラジオボタン モーダルダイアログ アコーディオン パンくずリスト テーブル カード ページナビゲーション ヘッダーコンテナ 検索ボックス 日付ピッカー ステップナビゲーション … 全40種以上
📌 デジタル庁デザインシステムとは
主に行政機関・公共機関での利用を念頭に構築されており、アクセシビリティの確保を優先した作りになっています。 行政サービスの画面で利用しやすいルールがそろっており、手続きのオンライン化などのプロジェクトで活用することで作業の効率化が図れます。 行政・公共機関に限らずどなたでも無料で利用可能です。

05 — Summary

まとめ：全体の関係性

概念 一言で言うと Figmaでの実現手段
デザインシステム デザインのルールブック＋部品集 ライブラリファイル全体
デザイントークン デザインの値に名前をつけて管理 バリアブル（Variables）
スタイル 複数の値をセットで管理 スタイル（Styles）
コンポーネント 再利用できる部品 コンポーネント（Components）
